# 临时下载站 A1 双栏 UI 改版设计

## 背景与目标

当前首页把文件、磁盘统计、aria2 任务、添加链接和本地上传依次纵向排列。功能完整，但朋友访问时最常用的文件浏览与下载不够突出，管理员操作也会拉长页面。

本次采用已确认的 A1 方案：桌面端左侧为管理员工具，右侧为更宽的文件主区域；移动端将管理员工具折叠为独立入口。目标是让普通访问者进入页面后立即找到文件、预览和下载，同时完整保留 v0.8.2 功能。项目继续使用 Python 标准库、服务端渲染 HTML、CSS 和少量原生 JavaScript，不引入框架、数据库或账号系统。

## 设计原则

- 文件优先：首屏和最大可用宽度交给文件列表。
- 管理操作归位：链接下载、本地上传和 aria2 状态集中在左侧。
- 保持轻量：不增加新页面框架、构建工具或前端依赖。
- 渐进增强：核心链接和表单仍由服务端 HTML 提供，JavaScript 只负责筛选、菜单、主题、二维码和分片上传体验。
- 功能不减：视觉收纳不等于删除功能，移动端也必须能访问完整表单和任务操作。

## 已选方案与取舍

### 方案 A1：固定左侧管理栏 + 右侧文件区（采用）

桌面端采用约 `300px minmax(0, 1fr)` 的两列网格。左栏容纳管理员工具，右栏容纳状态条、检索筛选和文件列表。它符合常见后台工具的阅读方向，也能让文件区获得剩余宽度。

### 顶部管理抽屉

可为文件区保留更多横向空间，但在桌面端会隐藏常用上传入口，任务状态也不便持续查看。本项目管理者人数少、操作频率稳定，因此不采用。

### 文件列表与管理页拆分

结构最清晰，但增加页面跳转和独立状态维护，不符合小型分享站“一页完成”的初衷，因此不采用。

## 页面架构

`render_home()` 仍提供单页首页，并沿用 `page()` 注入公共样式和脚本。首页结构调整为：

1. 顶部栏：站点名称、刷新按钮、主题切换按钮和现有下载目录入口。
2. 主体网格：左侧管理员栏，右侧文件主区域。
3. 全局层：二维码弹窗、Toast 和文件动作菜单。

不修改 `/downloads/`、`/view/`、`/file/`、`/once/` 及现有 `/api/*` 路由。预览页和下载目录页继续使用公共主题样式，只做必要的一致性调整，不重构其信息架构。

## 顶部栏

标题保持简洁，不使用营销式大标题。右侧使用图标按钮承载刷新与主题切换，图标约 `20px`，按钮保持稳定的点击区域并提供 `title`/`aria-label`。下载目录入口继续保留文字与图标，避免隐藏现有导航能力。

移动端标题与操作保持同一行；空间不足时允许操作组换行，但不得覆盖标题。

## 左侧管理员栏

管理员栏按工作流由上到下排列三个区块。区块使用紧凑分隔和小标题，不嵌套装饰性卡片。

### 添加链接

保留完整字段：HTTP/HTTPS URL、自定义文件名、保留时间、管理员密码和提交按钮。字段改为单列排列，帮助文字压缩为与输入相关的短提示。现有 `/api/add-task` 表单协议不变。

### 上传本地文件

保留拖放/文件选择、自定义文件名、保留时间、管理员密码和上传按钮。拖放区使用约 `28px` 文件图标，并显示选中文件名。

v0.8.2 上传状态必须完整保留：总进度、已上传容量、分片完成数、活跃 worker 数、重试次数和取消按钮。上传期间禁用提交按钮；取消和终态提示继续使用现有 `upload_cancel` 流程。失败信息显示在上传区内，不让用户依赖控制台判断结果。

### aria2 任务

默认显示任务数量和总体状态摘要，可展开查看任务名、状态、进度、速度、已下载/总大小和 GID。每个任务保留输入管理员密码后移除的能力，并保留清理已停止任务记录的表单。

侧栏在长页面中可使用 `position: sticky`，但自身高度必须受视口限制并允许内部滚动，避免任务较多时遮挡页面底部。

## 右侧文件主区域

### 紧凑状态条

现有四张大统计卡合并为一条紧凑状态带，展示文件数量、下载目录占用、剩余空间和默认保留时间。磁盘占用通过短进度条及文字共同表达，不能只依赖颜色。危险阈值继续沿用现有后端计算结果。

### 搜索与类型筛选

搜索框放在文件区标题下方，类型筛选使用紧凑分段按钮或筛选按钮组。继续基于已有 `data-name` 和 `data-type` 在浏览器端组合筛选，不新增 API。

筛选结果为空时显示“没有匹配的文件”；站点本身没有文件时显示独立的空状态，避免两种情况混淆。

### 文件行

每行使用约 `28px` 的类型图标，随后是文件名及次要元数据。桌面端显示大小、添加时间/剩余时间等现有信息；窄屏将次要元数据并入文件名下方，主操作始终可见。

操作规则：

- 可预览类型（图片、视频、文本）显示“预览”和“下载”。视频不得只剩预览。
- 不可预览类型显示“下载”。
- 更多 `...` 菜单包含二维码分享、一次性下载和续期。
- 续期继续使用现有管理员密码交互和后端路由，不在前端保存密码。
- 一次性下载属于破坏性语义，菜单中使用警示说明，但不增加新的二次确认流程。

菜单在任一时刻只打开一个；点击菜单外、按 `Escape` 或选择动作后关闭。菜单按钮具有 `aria-expanded` 和关联菜单标识，键盘焦点能够进入菜单项。二维码弹窗继续生成普通下载链接的二维码，并保留复制链接能力。

## 响应式与移动端

在约 `900px` 以下，双栏切换为单栏。文件主区域排在前面，页面顶部提供“管理员工具”入口，展开后显示完整的添加链接、上传和 aria2 区块。这里仅改变呈现顺序，不删减字段或操作。

移动端文件列表采用堆叠行而不是横向溢出的表格：图标、文件名和元数据占据上部，预览/下载及 `...` 位于下部或右侧稳定操作区。按钮文字可换行但不能溢出；图标点击区域至少约 `40px`。

二维码弹窗宽度受视口约束，动作菜单在靠近屏幕边缘时向内对齐。打开管理员工具或菜单不锁死页面滚动。

## 视觉样式

整体保持安静、实用和适合重复操作的工具界面。采用中性浅背景、白色内容面、深色正文、克制的蓝色主色，并用绿色/琥珀色/红色表示状态。避免大面积单一紫色、渐变装饰、过度圆角和卡片套卡片。

主要圆角不超过 `8px`。文件类型通过现有字符/图标方案或轻量内联图形表达，不引入图标依赖；图标大小统一约 `28px`，操作按钮图标约 `18-20px`。深色主题继续支持系统偏好和手动切换，所有文本、边框、焦点和危险状态需保持足够对比度。

## 数据与状态流

服务端继续通过 `scan_files()`、`get_disk_stats()` 和 `get_aria2_tasks()` 生成首页数据。改版仅重新组织其 HTML，不新增持久化字段。

上传仍按以下流程运行：

1. 表单调用 `POST /api/upload_init` 获取服务端会话、分片大小和并发数。
2. worker 队列调用 `POST /api/upload_chunk` 并汇总进度、活跃数和重试数。
3. 全部分片完成后调用 `POST /api/upload_finish`。
4. 用户取消时中止活动 XHR，并调用 `POST /api/upload_cancel`。

TTL 清理、容量预留、连接关闭策略以及 `read_ms`、`write_ms`、`rename_ms` 日志均保持后端行为，不在 UI 中暴露内部细节。

## 错误与空状态

- 服务端重定向回首页的消息继续显示为页面通知。
- JavaScript 上传错误显示在上传区，并恢复可再次提交的状态。
- aria2 RPC 不可用时只影响任务区，文件列表和上传表单仍可使用。
- 菜单和弹窗的脚本失败不应破坏普通 `<a>` 下载链接。
- 长文件名使用换行和断词策略，不截断到无法区分文件；操作区保持固定宽度，避免被文件名挤出视口。

## 测试与验收

### 自动化测试

在现有测试基础上增加首页渲染断言，至少覆盖：

- 双栏主体、左侧管理员区和移动端管理员入口存在。
- 添加链接与上传表单的全部字段和原有 action 保留。
- v0.8.2 四个上传端点、worker 并发、重试、状态文字和取消控件仍存在。
- aria2 任务移除与清理已停止记录的表单仍存在。
- 视频文件行同时包含 `/view/` 预览和 `/file/` 普通下载。
- `...` 菜单包含二维码分享、`/once/` 和续期入口。
- 搜索、类型筛选、主题切换和刷新控件存在。
- 用户提供的文件名、消息和任务信息继续 HTML 转义。

运行完整的 Python 测试，并对渲染出的内联 JavaScript 做语法检查。

### 视觉与交互验收

使用桌面和移动端视口检查首页，至少覆盖约 `1440x900`、`1024x768` 和 `390x844`：

- 桌面端左栏在左、文件区更宽，首屏能看到文件操作。
- 移动端文件区优先，管理员入口可展开全部功能。
- 长中英文文件名、空列表、筛选无结果、上传中和 aria2 错误状态无重叠或横向溢出。
- 视频行可预览也可下载，更多菜单可通过鼠标和键盘关闭。
- 浅色、深色主题下的焦点、状态和危险操作均清晰。

## 非目标

本次不增加数据库或数据库字段，不增加账号、提取码、服务端批量 ZIP、网盘能力、上传断点续传、文件批量管理或新的后端 API。若后续确实需要数据库字段扩充，必须先向用户确认。
