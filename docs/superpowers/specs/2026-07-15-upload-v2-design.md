# v0.8.2 Upload Pipeline v2 Design

## Context

The current browser splits a file into 5 MiB chunks and uploads up to three
chunks concurrently. Each chunk becomes a separate `.partN` file. The finish
request then reads every part and writes a second complete temporary file before
renaming it. This delays completion, temporarily approaches twice the file size
on disk, and provides too little information to validate or diagnose an upload.

This release keeps the project constraints: Python standard library, server-side
rendered pages, small JSON state files, no database, and suitability for a
low-memory Linux host with limited disk capacity.

## Scope

v0.8.2 will add:

- Server-created upload sessions.
- `upload_init`, `upload_chunk`, `upload_finish`, and `upload_cancel` endpoints.
- Offset writes into one temporary file instead of per-chunk part files.
- Bounded chunk retries in the browser.
- Expired session cleanup.
- A 50 MiB default limit for the legacy multipart fallback.
- Structured upload timing logs.
- Focused unit and HTTP integration tests.

The release will not add Range downloads, file deletion, retention fixes for
aria2, gallery views, resumable uploads after a browser restart, or database
storage. Those remain later-version work.

## Considered Approaches

### 1. Keep part files and improve finish

This is the smallest change, but finish still performs a full-file copy and the
temporary disk footprint remains close to double. Better logging would not fix
the underlying write amplification. Rejected.

### 2. Concurrent random writes to one temporary file

This can maximize simultaneous disk writes, but it requires careful coordination
between file writes and session-state updates. The expected bottleneck is the
network or tunnel, not serialization of short disk writes. The added race risk is
not justified for the first version. Rejected for v0.8.2.

### 3. Server sessions with serialized offset writes

Each browser request still uploads independently, but writes for one upload ID
are protected by a per-session lock. A chunk is read in bounded buffers and
written directly at its calculated offset. This removes part files and finish
copying while keeping state changes straightforward. Selected.

## Configuration

The following environment variables will be added:

```text
UPLOAD_CHUNK_BYTES=5242880
UPLOAD_CONCURRENCY=3
UPLOAD_SESSION_TTL_SECONDS=21600
UPLOAD_FALLBACK_MAX_BYTES=52428800
```

Values will be parsed with the existing bounded integer configuration pattern.
The server remains authoritative: the browser uses values returned by init.

## Session Storage

Each active upload has a JSON file under `data/uploads/<upload_id>.json` and one
temporary data file under `downloads/.upload-<upload_id>.tmp`.

The session contains:

```json
{
  "upload_id": "server-generated-id",
  "upload_token": "server-generated-secret",
  "filename": "video.mp4",
  "size": 220200960,
  "chunk_size": 5242880,
  "total_chunks": 42,
  "retention_seconds": 86400,
  "created_at": 1780000000.0,
  "updated_at": 1780000000.0,
  "received_chunks": [0, 1, 2],
  "received_bytes": 15728640
}
```

The upload ID and token are generated by the server. Paths are derived only from
the validated ID; clients never supply a filesystem path. JSON writes use a
temporary file and atomic replace. A process-local lock protects each session's
data file and JSON state.

This is ephemeral JSON state, not a database or a database schema expansion.

## API Design

### `POST /api/upload_init`

Accept form-urlencoded fields: `password`, `filename`, `custom_filename`,
`size`, `retention`, and optionally the browser's proposed `chunk_size`.

The server validates the password, final filename, positive size, per-file
limit, target collision, retention value, downloads quota, temporary upload
usage, and filesystem free space. It creates the session and empty temporary
file, then returns JSON containing the upload ID, upload token, chunk size,
total chunks, and concurrency.

No temporary file or session remains after a rejected init.

### `POST /api/upload_chunk`

Accept headers `X-Upload-Id`, `X-Upload-Token`, and `X-Chunk-Index`, plus an
ordinary `Content-Length` body.

The server loads the session, authenticates the token, checks the index and exact
expected chunk length, and calculates `offset = index * chunk_size`. Under the
session lock it reads at most 1 MiB at a time from the request and writes to the
single temporary file at that offset.

The chunk is marked received only after the complete body has been written and
flushed. Re-uploading an already completed chunk is idempotent: a valid
same-length retry rewrites the same range and remains successful.

### `POST /api/upload_finish`

Accept the upload ID and token. The server requires all chunk indexes, verifies
the exact temporary file size and target nonexistence, renames the temporary
file to the final filename, updates `filemeta.json` under `META_LOCK`, removes
the session JSON, logs timings, and returns JSON success.

The rename and metadata update cannot be one filesystem transaction. If rename
succeeds but metadata writing fails, the final file is retained and the error is
logged; a later `scan_files()` can reconstruct default metadata. The server must
not delete successfully uploaded user data in this failure case.

### `POST /api/upload_cancel`

Accept the upload ID and token. Under the session lock, remove the temporary file
and session JSON and write a cancellation log. Repeated cancellation returns
success when the session is already absent, so browser cleanup is idempotent.

## Browser Flow

The existing upload form remains. Submission performs init, uses the returned
chunk size and concurrency, uploads chunks through a bounded worker queue, and
calls finish after every worker succeeds.

Each chunk may be retried twice after the initial attempt, with 1-second and
3-second delays. A terminal failure aborts outstanding XHR requests where
possible, reports the server error, and leaves the session available for TTL
cleanup. The user can explicitly cancel while uploading, which aborts XHRs and
calls `upload_cancel`.

Progress continues to be calculated from per-chunk XHR progress. The status also
shows completed chunks, active workers, and retry count. The UI does not claim
that v2 increases physical network bandwidth.

## Legacy Upload Fallback

`POST /api/upload` remains for small uploads when JavaScript is unavailable. If
its `Content-Length` exceeds `UPLOAD_FALLBACK_MAX_BYTES`, the server returns 413
before reading the request body. Existing streaming multipart behavior remains
otherwise unchanged.

## Cleanup And Capacity Accounting

Cleanup removes session JSON and its matching temporary file when `updated_at`
is older than `UPLOAD_SESSION_TTL_SECONDS`. It also retains compatibility cleanup
for old `.partN` files left by the previous version.

Capacity checks include completed downloads and all `.upload-*.tmp` files so an
unfinished upload cannot bypass the downloads quota. Init reserves logically by
considering active session declared sizes; chunk handling repeats the filesystem
free-space check before writing.

## Logging

`logs/upload.log` uses one-line key/value entries without passwords or upload
tokens:

```text
event=init upload_id=... name=video.mp4 size=220200960 chunks=42
event=chunk upload_id=... idx=12 bytes=5242880 read_ms=9800 write_ms=35 total_ms=9837
event=finish upload_id=... size=220200960 rename_ms=4 total_ms=12
event=cancel upload_id=... reason=user
event=cleanup upload_id=... reason=expired
```

Filename values are encoded or escaped so one filename cannot create fake log
lines. `read_ms` measures request-body reads, `write_ms` measures file writes,
and `total_ms` measures the endpoint operation.

## Error Handling

- Authentication failures return 403.
- Invalid IDs, indexes, sizes, lengths, or filenames return 400.
- Existing target files return 409.
- Size and fallback limits return 413.
- Missing sessions return 404, except idempotent cancel.
- Incomplete request bodies do not mark chunks complete.
- Unexpected exceptions are recorded in `error.log` and return a generic 500.
- Responses for the new API endpoints are JSON, allowing the browser to display
  server messages without parsing HTML.

## Testing And Acceptance

Tests will cover:

- Init authentication, filename, size, collision, and capacity validation.
- Server-generated IDs/tokens and atomic session persistence.
- Exact chunk-length validation and offset writes.
- Idempotent chunk retry.
- Finish rejection when a chunk is missing or file size is wrong.
- Successful finish rename and metadata creation.
- Cancel and expired-session cleanup.
- Legacy multipart rejection before reading an oversized body.
- Upload log timing fields and secret omission.
- Existing v0.8.1 behavior and legacy small-file upload regression coverage.

Required verification:

```text
python -m py_compile download_server.py
python -m unittest tests.test_download_server -v
```

Deployment acceptance uses 10 MiB and approximately 200 MiB files, verifies
cancel and stale cleanup, and compares `read_ms`, `write_ms`, and finish time.
